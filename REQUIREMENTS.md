# PDF→プレゼン動画 自動生成サービス 要件定義書

## 1. プロジェクト概要

### 1.1 サービス名（仮）
**Slide2Video**

### 1.2 コンセプト
スライドPDFファイルをアップロードするだけで、AIが台本を生成し、
音声合成と合成して、プレゼンテーション動画（MP4）を自動生成するWebサービス。

### 1.3 ターゲットユーザー
- 自分のAI APIキーを持っている個人・開発者
- スライド資料から手軽に解説動画を作りたい人

---

## 2. 機能要件

### 2.1 基本フロー（一気通貫型）

1. ユーザーがPDFアップロード + APIキー入力
2. フロントエンド（Cloudflare Pages）からバックエンド（VPS）へ送信
3. バックエンドで以下を順次実行:
   - PDF → ページ画像に変換（PyMuPDF）
   - 画像/テキスト → AI台本生成（Gemini / OpenAI）
   - 台本 → 音声合成（Edge-TTS 無料 / OpenAI TTS 有料）
   - 画像 + 音声 → 動画合成（FFmpeg）
4. 完成動画URLを返却
5. ユーザーがブラウザでダウンロード

### 2.5 インフラ移行計画

| フェーズ | 構成 | 条件 |
|---------|------|------|
| Phase A（現在） | ローカルPC + ngrok | 開発・デモ用 |

| Phase C（スケール時） | Google Cloud Run + GCS + Redis/Firestore | 同時処理2〜3件超、VPSのCPU/メモリ逼迫、運用負荷が問題になった段階 |

**移行時の対応事項:**

- Phase C: 一時ファイルの保存先をローカルディスク → GCS に変更、ジョブ管理をメモリ内 dict → Redis/Firestore に変更

### 2.6 画像最適化設計（F-13）

**課題:** 現在PDF各ページをPNG（無圧縮）で書き出しており、1枚あたり1〜3MB。AI APIに送信する際のトークン消費が大きく、無料枠を圧迫する。

**設計方針:**
- PDF解析時に2種類の画像を生成する:
  - **AI用画像（WebP）:** 台本生成AIに送信する用。WebP品質80%、幅1024pxにリサイズ。150〜350KB程度。
  - **動画用画像（PNG）:** FFmpegで動画レンダリングに使用する用。元の高解像度を維持。

**圧縮効果の比較:**

| 形式 | 品質設定 | 1枚あたりのサイズ | トークン消費（概算） |
|------|---------|-----------------|-------------------|
| PNG（現在） | 無劣化 | 1〜3 MB | 5,000〜10,000 |
| JPEG 80% | わずかに劣化 | 200〜500 KB | 2,000〜4,000 |
| WebP 80% | わずかに劣化 | 150〜350 KB | 1,500〜3,000 |

**期待効果:**
- 1スライドあたりのトークン消費が約1/3〜1/5に削減
- Gemini無料枠のTPM（25万トークン/分）制限に余裕ができる
- AI台本の品質には影響なし（文字・図表の認識に十分な解像度）

**対応状況（全プロバイダーでWebPサポート確認済み）:**

| プロバイダー | PNG | JPEG | WebP |
|------------|-----|------|------|
| Gemini | ✅ | ✅ | ✅ |
| OpenAI (GPT-4o) | ✅ | ✅ | ✅ |
| OpenRouter経由 | ✅ | ✅ | ✅ |

**実装箇所:** `backend/services/pdf_service.py` の `parse_pdf` 関数を修正。PyMuPDFでページをレンダリング後、Pillowで WebP 変換+リサイズを行う。

### 2.7 バッチ台本生成設計（F-14）

**課題:** 現在は1スライド=1 APIリクエストのため、10枚PDFで10リクエスト消費。Gemini無料枠（20 RPD）では1日2本しか作れない。

**設計方針:**
- 複数スライド（2〜5枚）のテキストと画像を1リクエストにまとめて送信
- AIに「スライド1の台本:」「スライド2の台本:」のように区切って出力させる
- レスポンスをパースして各スライドの台本に分割

**バッチサイズとリクエスト数の比較:**

| バッチサイズ | 10枚PDFのリクエスト数 | Gemini無料枠(20RPD)で1日の本数 |
|------------|---------------------|-------------------------------|
| 1枚（現在） | 10 | 2本 |
| 2枚 | 5 | 4本 |
| 3枚 | 4 | 5本 |
| 5枚 | 2 | 10本 |
| 全枚一括 | 1 | 20本 |

**注意事項:**
- バッチサイズが大きすぎると出力品質が不安定になるリスクがある
- 画像を含む場合、1リクエストのトークン数が増大するためTPM制限に注意
- F-13（画像最適化）との併用で、バッチ時のトークン消費を大幅に抑制可能
- 推奨はバッチサイズ3〜5枚。全枚一括はスライド枚数が少ない場合のみ
- フォールバック: バッチ生成が失敗した場合は1枚ずつの逐次生成に切り替え

**ユーザー向けコスト情報（UIに表示検討）:**

| 構成 | コスト | 10枚PDFで1日に作れる本数 |
|------|--------|------------------------|
| Gemini（無料）+ Edge-TTS | ¥0 | 2本（バッチ化で最大20本） |
| OpenRouter（無料50req/日）+ Edge-TTS | ¥0 | 5本（バッチ化で最大25本） |
| OpenRouter（$10チャージ）+ Edge-TTS | 約¥1,500 | 実質無制限 |

### 2.8 レート制限案内リンク設計（F-15）

**概要:** AIプロバイダー選択時に、該当プロバイダーのレート制限・利用状況確認ページへのリンクをSettingsForm内に表示する。

**表示場所:** APIキー入力欄の直下（補足テキストの次）

**表示条件:**

| プロバイダー | 表示テキスト | リンク先 |
|------------|------------|---------|
| Gemini | 無料枠のレート制限はプロジェクトにより異なります。 | https://aistudio.google.com/rate-limit |
| OpenRouter | 利用状況と制限は OpenRouter ダッシュボードで確認できます。 | https://openrouter.ai/activity |
| OpenAI | 利用状況は OpenAI ダッシュボードで確認できます。 | https://platform.openai.com/usage |

**目的:** ユーザーが自身の無料枠・レート制限を事前に把握し、生成回数の見通しを立てられるようにする。

**実装箇所:** `frontend/src/components/SettingsForm.tsx`

### 2.9 ボイスクローン設計（F-16）

**概要:** Qwen3-TTS-VC は 10〜20 秒の音声サンプルからユーザーの声を複製し、テキスト読み上げに使用できる。

| 方式 | 実装 | コスト | 備考 |
|------|------|--------|------|
| A: DashScope API（Alibaba Cloud） | REST API で voice enrollment → synthesis | ボイス作成 $0.01 + 合成は従量課金 | 最も簡単。推奨プロトタイプ |
| B: ローカル実行（HuggingFace モデル） | Qwen3-TTS-12Hz-1.7B-CustomVoice をサーバーで推論 | GPU サーバー費用のみ（≥4GB VRAM） | API費用ゼロだがインフラ要件高 |
| C: セルフホスト API | 方式B をコンテナ化して Cloud Run GPU 等で提供 | GPU インスタンス費用 | スケーラブルだが設計が複雑 |

**推奨:** Phase B で方式A（DashScope API）をプロトタイプ実装。Phase C で方式B へ移行検討。

**音声サンプル要件:**
- 形式: WAV / MP3 / M4A
- 長さ: 10〜20秒推奨（最低3秒）
- 品質: 24kHz以上、モノラル推奨、雑音の少ない環境で録音
- 言語: 日本語 / 英語 / 中国語 / 韓国語 / フランス語 ほか対応

**法的留意点:** 他人の声（声優等）を使用する場合、本人の明示的な同意・ライセンス契約が必須。

### 2.9 AIモデル自動更新機構（F-16）

**課題:** LLM/TTSモデルは頻繁にアップデートされ、古いモデルIDが廃止されると動作不能になる。

**設計方針:**
- モデル一覧を `models.json` 等の外部定義ファイルで管理
- バックエンドは `/api/models` エンドポイントでフロントエンドに選択肢を提供
- フロントエンドはハードコードせず、API から動的に取得
- `models.json` 更新時にバックエンド再起動なしで反映（ホットリロード）

**models.json 構成例:**
```json
{
  "ai_providers": {
    "gemini": { "models": [{"id": "gemini-2.5-flash", "label": "Gemini 2.5 Flash", "default": true}] },
    "openai": { "models": [{"id": "gpt-4o", "label": "GPT-4o", "default": true}] },
    "openrouter": { "models": [{"id": "google/gemini-2.5-flash", "label": "Gemini 2.5 Flash (Google)", "default": true}] }
  },
  "tts_providers": {
    "edge-tts": { "voices": [{"id": "ja-JP-NanamiNeural", "label": "Nanami（女性）", "default": true}] },
    "openai": { "voices": [{"id": "alloy", "label": "Alloy（中性的）", "default": true}] },
    "qwen-clone": { "voices": [] }
  }
}
